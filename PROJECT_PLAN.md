## План реализации проекта (backend записи к врачу на NestJS)

### 1. Базовая инфраструктура проекта

- **Окружение**
  - Настроить файл `.env` на основе `.env.example` (`DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASS`, `DB_NAME`, `JWT_SECRET`, `PORT` и т.п.).
  - Подключить и сконфигурировать `ConfigModule` (`@nestjs/config`), сделать его глобальным.

- **База данных и TypeORM**
  - Поднять PostgreSQL (локально или через Docker).
  - Настроить `TypeOrmModule.forRoot` с конфигом из переменных окружения.
  - Описать базовые сущности: `User`, `Doctor`, `Patient`, `Appointment`, при необходимости `Specialization`, `Clinic`.

- **Общие настройки приложения**
  - В `main.ts` подключить глобальные пайпы (`ValidationPipe` с `class-validator` / `class-transformer`).
  - Подключить Swagger (`@nestjs/swagger`) для автодокументации REST API.
  - Сформировать структуру модулей: `AuthModule`, `UsersModule`, `DoctorsModule`, `PatientsModule`, `AppointmentsModule` и т.п.

### 2. Аутентификация и пользователи

- **Модель пользователя**
  - Создать сущность `User` (TypeORM): `id`, `email`, `passwordHash`, `role` (`admin`, `doctor`, `patient`), дополнительные поля (имя, телефон и т.д.).
  - Хранить пароль только в виде хеша (через `bcrypt`).

- **Регистрация и логин**
  - DTO: `RegisterDto`, `LoginDto`.
  - В `AuthService` реализовать:
    - регистрацию (проверка уникальности email, хеширование пароля, сохранение пользователя);
    - логин (проверка пароля, генерация `accessToken` через `JwtService`).
  - В `AuthController` добавить эндпоинты `/auth/register`, `/auth/login`.

- **JWT-авторизация**
  - Настроить `JwtModule` (`@nestjs/jwt`) с секретом и сроком жизни токена из `.env`.
  - Реализовать `JwtStrategy` и `JwtAuthGuard`.
  - Применить `@UseGuards(JwtAuthGuard)` к защищённым эндпоинтам.
  - (Опционально) реализовать рефреш-токены.

- **Роли и доступы**
  - Использовать декоратор `@Roles(...)` и `RolesGuard` для ограничения доступа.
  - Ограничить доступ к эндпоинтам по ролям (`admin`, `doctor`, `patient`).

### 3. Доменные сущности и бизнес-логика

- **Доктора**
  - Создать сущность `Doctor` (один-к-одному с `User` или отдельная связь).
  - Поля: специализация, описание, стаж, расписание (или ссылка на отдельную таблицу расписаний).
  - Модуль `DoctorsModule`:
    - админские CRUD-эндпоинты для управления врачами;
    - публичные эндпоинты для списка врачей/фильтрации по специализации.

- **Пациенты**
  - Создать сущность `Patient` (связь с `User`).
  - Поля: дата рождения, пол, номер полиса, адрес и т.п.
  - Модуль `PatientsModule`:
    - эндпоинты для получения/обновления профиля пациента;
    - (опционально) админский просмотр списка пациентов.

- **Расписание и слоты**
  - Спроектировать модель расписания:
    - Вариант А: таблица `AppointmentSlot` (`doctorId`, `startTime`, `endTime`, `isBooked`).
    - Вариант Б: генерация слотов по шаблону расписания врача.
  - Модуль `ScheduleModule` (или часть `DoctorsModule`):
    - эндпоинты для получения свободных слотов врача.

- **Записи (appointments)**
  - Создать сущность `Appointment`: `id`, `doctor`, `patient`, `startTime`, `endTime`, `status` (`BOOKED`, `CANCELLED`, `COMPLETED`), `reason`, комментарии.
  - Модуль `AppointmentsModule`:
    - для пациента: получение свободных слотов, создание записи, отмена записи;
    - для врача: список своих записей, изменение статуса (проведён/отменён);
    - для админа: просмотр всех записей, фильтрация по врачу/пациенту/статусу.

### 4. Валидация, ошибки и безопасность

- **Валидация DTO**
  - Для всех входящих запросов описать DTO с `class-validator`.
  - Убедиться, что глобальный `ValidationPipe` включён с опциями `whitelist`, `forbidNonWhitelisted`.

- **Обработка ошибок**
  - (Опционально) добавить глобальный фильтр ошибок для единообразного формата ответов.
  - Для типичных ошибок (пользователь не найден, слот уже занят и т.п.) возвращать корректные HTTP-коды.

- **Безопасность**
  - Настроить CORS под нужные домены фронтенда.
  - (Опционально) добавить `helmet`, rate limiting.
  - Следить, чтобы в ответах не утекали чувствительные данные (пароли, секреты).

### 5. Документация API и контракт с фронтендом

- **Swagger**
  - Описать все эндпоинты декораторами `@ApiTags`, `@ApiOperation`, `@ApiResponse`.
  - Добавить `@ApiProperty` к DTO для генерации схем.
  - Настроить в Swagger схему авторизации (Bearer JWT).

- **Контракт с фронтендом**
  - Согласовать:
    - структуры DTO для запросов/ответов;
    - формат ошибок и коды ответов;
    - фильтры/сортировки для списков (врачи, записи).

### 6. Тестирование и качество кода

- **Юнит-тесты**
  - Написать тесты для ключевых сервисов (`AuthService`, `AppointmentsService` и т.д.) с использованием `@nestjs/testing` и `jest`.

- **E2E-тесты**
  - Настроить `test:e2e` (использовать пример из Nest-шаблона).
  - Описать сценарии: регистрация, логин, создание записи, отмена записи, проверка доступа по ролям.

- **Линтинг и форматирование**
  - Регулярно запускать `npm run lint` и `npm run format`.
  - При необходимости донастроить `eslint` и `prettier` под стиль проекта.

### 7. Нефункциональные задачи и деплой

- **Конфигурация под prod**
  - Разделить конфиги окружений (`.env.development`, `.env.production`).
  - Настроить логирование NestJS (минимальное в prod, расширенное в dev).

- **Docker (опционально)**
  - Создать `Dockerfile` для приложения.
  - Создать `docker-compose.yml` для приложения и PostgreSQL.
  - Настроить миграции TypeORM (если используются).

- **Деплой**
  - Выбрать платформу (VPS, Render, Railway, Mau и т.п.).
  - Настроить сборку (`npm run build`) и запуск `npm run start:prod`.
  - Прописать переменные окружения на прод-сервере.
